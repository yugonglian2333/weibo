## 一、数据模型设计

### 1.1 单个用户的数据结构

```yaml
# users.enc （AES 加密存储）
users:
  - id: "user_001"
    name: "张三"                          # 备注名
    cookie: "_s_tentry=...; SUB=..."      # 微博登录 Cookie
    cookie_updated: "2026-06-02"          # Cookie 获取时间
    
    # 签到配置
    checkin:
      topics:
        - name: "鞠婧祎超话"
          containerid: "1008086f1b..."
    
    # 发帖配置（可选，每个用户可不同）
    posting:
      enabled: true
      topics: ["鞠婧祎超话"]
      style: "自然随性"
    
    # AI 发帖开关（可选，用于差异化收费）
    ai_enabled: true
    
    # 通知配置（用户自己的通知渠道）
    notification:
      channel: "pushplus"
      token: "9a2ed91b..."
    
    # 管理字段
    status: "active"                      # active / paused / expired
    created: "2026-06-02"
    expired: "2026-07-02"                # 到期时间（月付）
```

### 1.2 全局配置（config.yaml，不加密）

```yaml
# 全局 AI 配置（你自己提供，所有用户共用）
ai:
  provider: mimo
  api_key: "tp-cswmq..."
  api_base: "https://..."
  model: "mimo-v2.5-pro"

# 管理员通知（你自己的 PushPlus，接收系统级通知）
admin_notify:
  channel: pushplus
  token: "你的token"
```

### 1.3 GitHub Secrets 精简

| Secret | 说明 | 变更频率 |
|--------|------|----------|
| `DATA_KEY` | AES 加密密码 | **永远不变** |
| `AI_API_KEY` | AI 服务商 Key | 偶尔 |
| `ADMIN_NOTIFY_TOKEN` | 你自己的通知 Token | 偶尔 |

> 就 **3 个 Secret**，加 1000 个用户也是这 3 个。不再需要为每个用户单独创建 7 个 Secret。

## 二、加密方案

### 2.1 整体架构

```
┌──────────────────────────────────────────────────────┐
│  本地（你的电脑）                                      │
│                                                       │
│  manage_users.py                                      │
│     │                                                 │
│     ├── add     → 扫码登录 → 抓 Cookie                │
│     │            → 配置话题/通知                      │
│     │            → AES 加密追加到 users.enc            │
│     ├── list    → 解密展示所有用户（Cookie 脱敏）       │
│     ├── remove  → 删除指定用户                        │
│     ├── pause   → 暂停某用户（到期不续费）             │
│     └── export  → 导出某用户明文 Cookie（调试用）      │
│                                                       │
│  users.enc  ←── Git 跟踪（密文，可安全提交到仓库）      │
│                                                       │
├──────────────────────────────────────────────────────┤
│  GitHub Actions（云端执行）                            │
│                                                       │
│  main.py --all                                        │
│     │                                                 │
│     ├── 从 Secrets 读取 DATA_KEY                      │
│     ├── 解密 users.enc                                │
│     ├── 过滤 status=active 的用户                     │
│     │                                                 │
│     └── 逐个用户执行（串行）:                          │
│          ├── WeiboClient(user.cookie)  ← 直接传 Cookie │
│          ├── 签到                                     │
│          ├── AI 生成 → 发帖（如果 ai_enabled）         │
│          └── 通知到用户自己的 PushPlus                 │
│                                                       │
├──────────────────────────────────────────────────────┤
│  管理通知                                             │
│                                                       │
│  全部执行完后：                                       │
│     └── 汇总报告 → 推送给管理员                        │
│        "今日: 15/18 用户签到成功, 12/12 发帖成功       │
│         失败用户: 张三(Cookie过期), 李四(限流)"         │
└──────────────────────────────────────────────────────┘
```

### 2.2 加密细节

- **算法**: AES-256-GCM（带认证加密，防篡改）
- **密钥**: 32 字节随机字符串，保存在 `DATA_KEY` Secret 中
- **文件格式**: JSON → AES 加密 → Base64 编码 写入 `users.enc`
- **安全保证**:
  - 密文可安全提交到 Git 仓库
  - Cookie 只在运行时内存中存在，不入环境变量
  - 日志中绝不打印 Cookie 原文（只打印用户备注名）

### 2.3 密钥生成

```bash
# 生成 32 字节随机密钥
python -c "import secrets; print(secrets.token_hex(32))"
# 输出示例: 8f3a91cb7e2d4609a5bf1c83e4726ab9...
```

将生成的密钥设为 GitHub Secret `DATA_KEY`，**千万不要丢失**——丢了所有用户数据就找不回来了。

## 四、main.py 改造要点

### 4.1 入口参数

```bash
python main.py              # 默认 --all，遍历所有 active 用户
python main.py --all        # 显式全部
python main.py --user 张三  # 只跑一个用户（调试用）
```

### 4.2 执行流程

```
1. 读取 DATA_KEY 环境变量
2. 解密 users.enc → 得到用户列表
3. 过滤 status == "active" 的用户
4. 检查是否有到期的用户（自动标记为 expired）
5. 初始化 Playwright 浏览器（所有用户复用同一个 browser）
6. 逐个用户执行:
   a. 创建新的 browser context，注入该用户的 Cookie
   b. 执行签到（checkin）
   c. 执行 AI 发帖（如果 ai_enabled）
   d. 通知该用户（如果配置了 notification）
7. 关闭浏览器
8. 汇总报告 → 通知管理员
```

### 4.3 安全要点

| 原则 | 实现 |
|------|------|
| Cookie 不入环境变量 | 解密后直接在代码中传给 `WeiboClient` |
| 日志不打印 Cookie | 只打印 `用户「张三」签到中...` |
| 单用户失败不影响其他 | try/except 包裹每个用户的执行 |
| 管理员通知不含敏感信息 | 只报告成功/失败数量，不含 Cookie |

## 五、GitHub Actions 执行流程

### 5.1 daily.yml 简化

```yaml
name: 微博每日签到发帖

on:
  schedule:
    - cron: '0 0 * * *'        # UTC 0:00 = 北京时间 8:00
  workflow_dispatch:

jobs:
  weibo-bot:
    runs-on: ubuntu-latest
    steps:
      - name: 检出代码
        uses: actions/checkout@v4

      - name: 设置 Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: 安装依赖
        run: |
          pip install -r requirements.txt
          playwright install chromium --with-deps

      - name: 执行多用户任务
        env:
          DATA_KEY: ${{ secrets.DATA_KEY }}
          AI_API_KEY: ${{ secrets.AI_API_KEY }}
          ADMIN_NOTIFY_TOKEN: ${{ secrets.ADMIN_NOTIFY_TOKEN }}
        run: python main.py --all
```

### 5.2 执行时序

```
每天 8:00 触发
    │
    ▼
检出代码
    │
    ▼
安装依赖 + Playwright Chromium
    │
    ▼
DATA_KEY 解密 users.enc
    │
    ▼
┌─────────────────────────────────┐
│  遍历 active 用户（串行）        │
│                                 │
│  张三:                          │
│    ├── 签到 2/2 ✅              │
│    ├── AI 发帖 2/2 ✅           │
│    └── 通知张三 ✅              │
│                                 │
│  李四:                          │
│    ├── 签到 2/2 ✅              │
│    ├── AI 发帖 1/2 ❌ (限流)    │
│    └── 通知李四 ✅              │
│                                 │
│  王五:                          │
│    ├── Cookie 过期 ❌           │
│    └── (不通知王五，只记日志)    │
│                                 │
└─────────────────────────────────┘
    │
    ▼
汇总报告 → 通知管理员
  "3 用户: 2 成功, 0 部分失败, 1 Cookie 过期"
```

### 5.3 时间与资源

- **单用户耗时**: ~30 秒（Playwright 复用 browser context）
- **20 用户**: 约 10 分钟
- **50 用户**: 约 25 分钟
- **GitHub Actions 限制**: 单次最长 6 小时，完全够用
- **免费额度**: GitHub 公开仓库无限分钟，私有仓库 2000 分钟/月

## 八、安全分析

### 8.1 威胁模型

| 威胁 | 风险等级 | 缓解措施 |
|------|----------|----------|
| users.enc 泄露（GitHub 公开仓库） | 低 | AES-256-GCM 加密，无 DATA_KEY 无法解密 |
| GitHub Secrets 泄露 | 中 | GitHub 对 Secrets 有访问控制，仅 Actions 运行时注入 |
| 日志泄露 Cookie | 中 | 日志仅打用户备注名，不打印 Cookie 原文 |
| DATA_KEY 丢失 | 高 | 备份在安全位置（1Password / 本地加密文件） |
| 用户自己修改了密码导致 Cookie 失效 | 中 | 管理员汇总通知中报告 Cookie 过期用户 |

### 8.2 关键原则

1. **DATA_KEY 是唯一命门**——妥善保管，离线备份
2. **Cookie 只在内存中存在**——不入环境变量，不入日志，不入文件
3. **users.enc 可以 Git 跟踪**——密文本身是安全的
4. **每个用户的通知独立**——不会泄露 A 的信息给 B

## 十、总结：单用户 vs 多用户

| 维度 | 当前（单用户） | 改造后（多用户） |
|------|---------------|-----------------|
| Cookie 存储 | `.env` 明文 | `users.enc` AES-256 加密 |
| Secrets 数量 | 7 个（每个环境变量） | **3 个**（永久不变） |
| 多用户支持 | ❌ 只能跑一套 | ✅ 遍历所有 active 用户 |
| 加新用户 | 改 `.env` + 同步 Secrets | `python manage_users.py add` → `git push` |
| 用户隔离 | 无 | 各自 Cookie + 各自通知渠道 |
| 收费支持 | 无 | `expired` 字段 + `status: paused` |
| 管理员监控 | 无 | 每日汇总通知（知道谁成功了谁失败了） |
| Cookie 获取 | 手动 F12 复制 或 `get_cookie.py` 覆盖 | `manage_users.py` 引导式扫码 |
